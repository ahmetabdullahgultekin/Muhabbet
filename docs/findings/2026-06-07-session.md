## The build failures (sequential) and how each was fixed

The Android debug build was failing; the failures surfaced one after another as each prior blocker
was cleared. Three blocked the app build (PR #49); a fourth blocked the mobile unit-test compile
(`claude/fix-tests`).

### 1. Firebase `-ktx` artifacts no longer exist (unresolved dependencies)
- **Symptom:** `firebase-auth-ktx` / `firebase-messaging-ktx` could not be resolved, and the imports
  `com.google.firebase.auth.ktx.auth` / `com.google.firebase.ktx.Firebase` were unresolved.
- **Root cause:** Firebase BoM `34.11.0` **dropped the separate `-ktx` artifacts** — the Kotlin
  extensions were folded into the main `firebase-auth` / `firebase-messaging` modules.
- **Fix:** In `mobile/composeApp/build.gradle.kts`, switched
  `com.google.firebase:firebase-auth-ktx` → `firebase-auth` and
  `firebase-messaging-ktx` → `firebase-messaging`. Updated `FirebasePhoneAuth.android.kt` imports to
  `com.google.firebase.auth.auth` and `com.google.firebase.Firebase`.

### 2. `compileSdk` too low
- **Symptom:** After the dependency resolved, the build required a newer compile SDK than `35`.
- **Root cause:** An AndroidX/Firebase-era dependency required compiling against API 36.
- **Fix:** `compileSdk = 35` → `36` in `mobile/composeApp/build.gradle.kts`.

### 3. libsignal Signal Protocol code does not compile against the pinned API
- **Symptom:** The Android Signal Protocol sources fail to compile.
- **Root cause:** `SignalKeyManager.kt`, `SignalEncryption.kt`, `InMemorySignalProtocolStore.kt`, and
  `PersistentSignalProtocolStore.kt` are written for a **≤0.70-era libsignal API**, but the module is
  pinned at `org.signal:libsignal-android:0.86.5` (and newer versions are only on Signal's own Maven
  repo). The API moved underneath the code: `Curve` was removed (~0.76), `IdentityKeyStore.saveIdentity`
  now returns `IdentityChange` (~0.73), `PreKeyBundle` requires the 11-arg Kyber form, and
  `SessionCipher` requires a `localAddress` arg (since 0.91). Full detail: `CLAUDE.md` →
  "libsignal upgrade (BLOCKED)".
- **Fix (temporary, build-unblock only):** Renamed the 4 Signal files to `*.kt.disabled` so they are
  excluded from compilation, and changed `PlatformModule.android.kt` to wire `NoOpKeyManager()` +
  `NoOpEncryption()` (the same NoOp path iOS already uses) instead of the Signal implementations.
- **Consequence — important:** This makes E2E a **NoOp placeholder that returns plaintext (NOT
  secure)**. It is byte-identical to current prod behavior **only because** E2E is flag-OFF by default
  (`E2EConfig.ENABLED = false`). E2E must remain OFF: flipping the flag now would send plaintext
  labelled as encrypted. **Re-integrating the libsignal API is the standing blocker** to restore real
  E2E; do not re-enable the disabled files until that rewrite lands + is crypto-reviewed.

### 4. Ktor 3.x moved `HttpResponseData` (mobile unit-test compile failure)
- **Symptom:** `:mobile:composeApp:testDebugUnitTest` failed with "Unresolved reference
  HttpResponseData", failing the whole mobile unit-test suite.
- **Root cause:** `AuthRepositoryTest.createApiClientWithMock` referenced
  `io.ktor.client.engine.mock.HttpResponseData`, but in **Ktor 3.x** that type lives in
  `io.ktor.client.request.HttpResponseData` — the `mock` package now only holds `MockRequestHandleScope`.
- **Fix:** Pointed the mock handler's lambda return type at `io.ktor.client.request.HttpResponseData`
  (one-line, test-only change on branch `claude/fix-tests`).

## Splash "green circle" — root cause

- **Symptom:** A stray green circle overlapped the login screen on launch.
- **Root cause:** `splash_background.xml` / `splash_background_dark.xml` used a `layer-list` that drew
  a centered **green oval** (`#1B5E20`, 120dp). Because the Compose content is transparent and never
  paints an opaque background, the window-background oval **bled through** the first composable and
  rendered as a stray green circle over the login UI.
- **Fix:** Flattened the drawable to a plain solid window background (a flat-color `shape` rectangle,
  no oval layer). A comment in the file documents why it must stay a flat color.

## Login — root cause and the fallback

- **Root cause (two compounding factors):**
  1. **Firebase phone-auth is API-key-restricted** on the current build, so Firebase verification
     fails with a configuration/internal error ("API key not valid" class of failures), not just
     transient throttling.
  2. The **prod backend OTP sender is still the mock** (`MockOtpSender`) — so even the backend path
     doesn't send a real SMS in production today.
- **Fix shipped (client fallback):** `PhoneInputScreen.kt` now routes Firebase failures through a new
  `shouldFallbackToBackendOtp(rawMessage)` helper. The previous logic only caught **rate-limiting**
  phrases ("block" / "too many" / "unusual"); the new helper also catches **configuration/internal**
  errors ("api key", "not valid", "internal error", "configuration", "developer error",
  "app not authorized", "invalid-api-key"). So a misconfigured-Firebase build degrades gracefully to
  the backend OTP flow instead of dead-ending on a raw error. Firebase remains the primary path, and
  user-facing errors now resolve to the localized `phone_auth_failed` string (TR+EN added).
- **Still required for prod (tracked in `TODO.md` P0):** wire a real SMS sender (Twilio/Netgsm) on the
  backend and verify FCM push end-to-end — only then can a real user actually log in and be notified.
  The client fallback makes the path resilient, but the backend must produce a real OTP for it to help.

## The three feature / UI slices

### Scheduled-message send UI (`claude/feat-scheduled-send-ui`, PR #50)
- Long-press the send button opens a two-step Material3 date + time picker; the chosen epoch-millis is
  set as `scheduledAt` on the existing `WsMessage.SendMessage` frame (the field and the backend
  `ScheduledMessageJob` already existed). A normal tap still sends immediately.
- A "Scheduled" chip above the input bar surfaces session-pending scheduled messages with per-item
  cancel (reuses `GroupRepository.deleteMessage`, so the backend job won't deliver a cancelled item).
- No business logic in composables: send via `WsClient`, cancel via the repository, date/time math via
  `DateTimeFormatter`. New `ScheduledSend.kt`; edits to `ChatScreen.kt`, `MessageInputPane.kt`,
  `DateTimeFormatter.kt`; 12 `schedule_*` strings (TR+EN). Android-verified.

### Communities — real group-picker sheet (`claude/feat-communities-add-group`, PR #51)
- Replaced the "coming soon" stub in `CommunityDetailScreen` with an `AddGroupToCommunitySheet`
  `ModalBottomSheet` that lists the user's `GROUP` conversations (excluding groups already in the
  community) and calls the existing `CommunityRepository.addGroupToCommunity()` endpoint, then
  refreshes the detail. Dropped unused `coming_soon`/`ok` strings; added subtitle/empty/added/failed
  strings (TR+EN).

### Mute-duration picker cleanup (`claude/feat-mute-duration-ui`, PR #52)
- `MutePickerDialog` previously rendered each duration (8h / 1w / Always) with a permanently-unselected
  `RadioButton` whose `onClick` duplicated the row's own clickable handler — a misleading affordance
  and a redundant tap target. Replaced with plain clickable rows carrying a leading `NotificationsOff`
  icon, `MuhabbetSpacing` tokens, `onSurfaceVariant` tint, and a 48dp (`MuhabbetSizes.MinTouchTarget`)
  minimum height. **No behaviour change** — still calls `onMuteDuration(key)` →
  `ConversationRepository.muteConversation(id, duration)`.
